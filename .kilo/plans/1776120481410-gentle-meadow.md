# Plan de implementación: jerarquía multi-tenant + validación centralizada

## Objetivo

Reestructurar autenticación, autorización y primera experiencia de acceso para que el sistema refleje esta jerarquía funcional:

1. `super_admin_global`: Sosercom/founders. Control total del sistema y validación superior.
2. `admin`: estudio jurídico o abogado principal que contrata el servicio.
3. Roles dependientes del tenant del estudio: `abogado`, `contador`, `tributario`, `staff`.
4. `cliente`: cliente final administrado dentro del tenant del estudio.

La habilitación operará por estado de cuenta:

- `pending_validation`: autenticado pero bloqueado en una vista intermedia.
- `active`: acceso operativo a su zona autorizada.
- `suspended`: acceso autenticado pero sin operación normal.

## Estado actual detectado

- `src/lib/auth/AuthContext.tsx` ya carga `role`, `status`, `tenantId`, `companyId`, `department` desde Firestore, pero mantiene el rol legado `admin_interno` y fallback de `status` a `active`.
- `src/components/auth/ProtectedRoute.tsx` redirige por rol de forma parcial y luego delega a `ValidationGuard`.
- `src/components/auth/ValidationGuard.tsx` ya muestra una vista de espera para `pending_validation`, pero solo cubre ese estado y no enruta por contexto de negocio.
- `firestore.rules` mezcla la lógica nueva de `super_admin_global` con el rol antiguo `admin`, y además permite `write` a cualquier usuario sobre su documento completo, lo que deja huecos para elevar privilegios o alterar su propio `status`.
- `src/app/login/page.tsx` registra auditoría con rol fijo `cliente` y redirige siempre a `/dashboard`, ignorando el rol real.
- Layouts y sidebar siguen usando `admin_interno`; la navegación aún no expresa claramente la jerarquía superadmin -> tenant admin -> equipo -> cliente.
- Algunas pantallas cliente (`src/app/dashboard/documentos/page.tsx`) siguen contemplando roles internos antiguos dentro del dashboard de cliente.

## Decisiones de arquitectura

### 1. Fuente única de verdad para identidad

Consolidar el modelo de usuario en Firestore como base de autorización de frontend y reglas:

Campos mínimos por `users/{uid}`:

```ts
type UserRole =
  | "super_admin_global"
  | "admin"
  | "abogado"
  | "contador"
  | "tributario"
  | "staff"
  | "cliente";

type UserStatus = "pending_validation" | "active" | "suspended";

interface UserDoc {
  uid: string;
  email: string;
  displayName: string | null;
  role: UserRole;
  status: UserStatus;
  tenantId: string | null;
  companyId: string | null;
  department: string | null;
  createdBy: string | null;
  validatedBy: string | null;
}
```

Regla operativa:

- Solo `super_admin_global` valida altas globales y activa tenants principales.
- `admin` administra usuarios y clientes de su tenant, sin capacidad de convertirse en superadmin ni de alterar validaciones globales sensibles.
- `cliente` queda siempre acotado a su empresa (`companyId`) y tenant (`tenantId`).

### 2. Enrutamiento por zona y no solo por rol aislado

Definir zonas explícitas:

- Zona `super-admin`: `/super-admin/*`
- Zona `admin`: `/admin/*`
- Zona `cliente`: `/dashboard/*`

Mapa recomendado:

- `super_admin_global` -> acceso a todas las zonas, landing interna por defecto `/super-admin`
- `admin`, `abogado`, `contador`, `tributario`, `staff` -> zona `/admin`
- `cliente` -> zona `/dashboard`

Esto simplifica `ProtectedRoute`, evita condicionales repetidos y permite una función única del tipo `getDefaultRouteForRole(role)`.

### 3. Validación por estado antes de entrar a cualquier zona protegida

Orden recomendado del guard:

1. Sin sesión -> `/login`
2. Con sesión pero sin documento de usuario válido -> logout + `/login`
3. `pending_validation` -> pantalla dedicada de espera
4. `suspended` -> pantalla dedicada de bloqueo/soporte
5. Rol sin acceso a la zona actual -> redirección a su zona por defecto
6. `active` + rol válido -> render normal

Esto deja a `ProtectedRoute` como puerta única y hace innecesarias excepciones dispersas en páginas internas.

### 4. Reglas Firestore herméticas por capacidad

Las reglas deben impedir explícitamente:

- autoasignarse un rol superior
- cambiar el propio `status`
- cambiar `tenantId`, `companyId` o `validatedBy` sin permisos
- leer documentos fuera del tenant o fuera de la empresa correspondiente

En vez de `allow write: if request.auth.uid == userId`, separar `create`, `update`, `delete` y validar campos modificables.

## Cambios a implementar

### Backend lógico / autenticación

#### `src/lib/auth/AuthContext.tsx`

- Reemplazar `admin_interno` por `admin` en el tipo de rol.
- Eliminar fallback silencioso de `status` a `active`; el valor por defecto seguro debe ser `pending_validation` si falta en Firestore.
- Añadir helpers exportables:
  - `isAdminRole(role)`
  - `isClientRole(role)`
  - `getDefaultRouteForRole(role)`
  - `canAccessRoute(role, pathname)` o equivalente
- Mantener el contexto como fuente reutilizable para layouts, header y sidebar.

#### `src/components/auth/ProtectedRoute.tsx`

- Unificar autenticación, validación por estado y autorización por zona.
- Sustituir redirecciones hardcodeadas por helpers basados en rol.
- Evitar decisiones especiales por pathname salvo las indispensables para detectar la zona actual.
- Mostrar loader mientras se resuelve la identidad.

#### `src/components/auth/ValidationGuard.tsx`

- Convertirlo en componente de estado de cuenta, no solo “pending”.
- Soportar al menos:
  - `pending_validation`: mensaje de revisión por Sosercom / estudio responsable.
  - `suspended`: mensaje de cuenta suspendida con salida segura.
- Ajustar el copy a la jerarquía nueva: Sosercom valida arriba; luego el tenant administra su estructura delegada.

#### `src/app/login/page.tsx`

- Dejar de auditar con rol fijo `cliente`.
- Tras login exitoso, esperar resolución del contexto o resolver documento del usuario para enviar a la ruta por defecto correcta.
- Si la cuenta está `pending_validation`, permitir que el flujo llegue a la vista intermedia, no a `/dashboard` por defecto.

### Frontend / estructura de navegación

#### Layouts protegidos

- `src/app/super-admin/layout.tsx`: mantener solo `super_admin_global`.
- `src/app/admin/layout.tsx`: actualizar a `admin`, `abogado`, `contador`, `tributario`, `staff`, y opcionalmente permitir `super_admin_global` por supervisión.
- `src/app/dashboard/layout.tsx`: restringir a `cliente`, o abrirlo solo si después se decide que ciertos roles internos pueden entrar en modo vista cliente. Recomendación actual: solo `cliente`.

#### `src/components/layout/sidebar.tsx`

- Reemplazar la lógica por secciones visibles según rol real.
- No mostrar “Mi Panel” a roles internos si no usarán `/dashboard`.
- Mostrar “Administración” a `admin` y dependientes internos.
- Mostrar “Super Admin” solo a `super_admin_global`.
- Mantener logout, pero centralizar navegación consistente con `getDefaultRouteForRole`.

#### `src/components/layout/header.tsx`

- Consumir `useAuth()` para avatar, iniciales, nombre/rol si procede.
- Corregir el item “Cerrar Sesión” para que realmente ejecute logout.
- Título por pathname puede mantenerse, pero conviene no depender de labels ambiguos heredados.

### Reglas Firestore

#### `firestore.rules`

Refactor completo recomendado con helpers:

- `isAuthenticated()`
- `currentUser()`
- `isSuperAdmin()`
- `isTenantAdmin()` usando `role == 'admin'`
- `isTenantMember(tenantId)`
- `isSameCompany(companyId)`
- funciones de validación de cambios de perfil para evitar escalamiento

Reglas mínimas:

- `users/{userId}`
  - `read`: superadmin, el propio usuario, o admin de su tenant según necesidad operativa.
  - `create`: superadmin y posiblemente `admin` solo para usuarios de su mismo tenant y sin poder asignar `super_admin_global`.
  - `update`: el propio usuario solo en campos de perfil inocuos; superadmin en todo; `admin` solo en campos permitidos para miembros de su tenant.
  - `delete`: superadmin, y opcionalmente `admin` sobre usuarios no privilegiados de su tenant.
- `companies/{companyId}`
  - lectura por tenant correspondiente
  - escritura por superadmin y `admin` del tenant
- `audit_logs/{id}`
  - escritura autenticada controlada según modelo actual
  - lectura solo superadmin, o si hace falta admin con filtrado de tenant en una fase posterior
- cualquier colección cliente (`documents`, etc.)
  - lectura/escritura filtrada por `tenantId` y/o `companyId`

Nota: Firestore rules no filtra campos de respuesta; por eso conviene asumir que datos sensibles de superadmin no se exponen en documentos legibles por admins.

### Primera interfaz requerida

Construir una pantalla de cuenta pendiente que sea la primera interfaz del flujo protegido para cuentas no activas.

Objetivos UX:

- explicar claramente que el acceso fue recibido
- indicar quién valida el acceso
- diferenciar si la cuenta pertenece a un estudio/abogado o a un cliente final
- ofrecer salida segura (`logout`)
- mantener coherencia visual con el portal actual

Contenido recomendado:

- título: “Tu cuenta está en validación”
- subtítulo contextual según rol
- bloque con pasos siguientes
- bloque de estado: “Registro recibido”, “Verificación en curso”, “Acceso operativo al aprobarse”

## Secuencia de ejecución recomendada

1. Normalizar modelo de roles y helpers en `AuthContext`.
2. Reescribir `ProtectedRoute` para que gobierne sesión, estado y ruta por rol.
3. Expandir `ValidationGuard` a estados de cuenta reales.
4. Corregir `login/page.tsx` para redirigir por rol y auditar con usuario real.
5. Actualizar layouts protegidos (`super-admin`, `admin`, `dashboard`).
6. Actualizar `sidebar` y `header` para reflejar la jerarquía nueva.
7. Endurecer `firestore.rules` con permisos por tenant y campos inmutables.
8. Revisar páginas cliente que aún aceptan roles internos heredados, empezando por `src/app/dashboard/documentos/page.tsx`.
9. Ejecutar validación funcional y lint.

## Impacto esperado

- Menos ambigüedad entre dueño del sistema, estudio jurídico y cliente final.
- Menor densidad mental en frontend al separar zonas por rol.
- Menor riesgo de escalamiento de privilegios en Firestore.
- Primer flujo de onboarding listo para cuentas pendientes antes de construir más pantallas.

## Validaciones posteriores a la implementación

- Login con `super_admin_global` -> entra a `/super-admin`.
- Login con `admin` activo -> entra a `/admin`.
- Login con `abogado`/`contador`/`tributario`/`staff` activo -> entra a `/admin`.
- Login con `cliente` activo -> entra a `/dashboard`.
- Login con cualquier rol `pending_validation` -> ve pantalla intermedia, sin acceso a módulos.
- Usuario autenticado intentando otra zona (`cliente` a `/admin`, `admin` a `/dashboard`) -> redirección a su home correcta.
- Intento de modificar `role` o `status` desde cliente Firestore -> denegado por reglas.

## Ejecución en Firebase y GitHub

Cuando se salga de plan mode, la ejecución debe incluir también estas acciones operativas:

1. Aplicar cambios de código local.
2. Desplegar reglas con `firebase deploy --only firestore:rules` usando el proyecto correcto.
3. Si hay cambios de documentos semilla o usuarios demo, actualizarlos manualmente en Firebase Auth / Firestore para reflejar:
   - reemplazo de `admin_interno` por `admin`
   - presencia explícita de `status`
   - `tenantId` y `companyId` consistentes
4. Validar login real con al menos un usuario por cada zona (`super_admin_global`, `admin`, `cliente`).
5. Ejecutar `npm run lint` y, si aplica, `npm run build`.
6. Crear commit con mensaje coherente con el cambio.
7. Push al remoto GitHub de la rama de trabajo.

### Prerrequisitos para esa ejecución

- Confirmar el proyecto Firebase objetivo para el deploy de reglas.
- Confirmar si el push debe ir a la rama actual o a una rama nueva para PR.
- Confirmar si además del push hay que abrir PR o solo dejar los cambios subidos.

## Riesgos identificados

- Hay inconsistencia documental entre README/resumen y código actual (`admin_interno`, `admin`, `staff`), por lo que conviene actualizar la documentación una vez terminada la implementación.
- `logAuditAction` hoy depende de un `AppUser`; si el login audita antes de resolver el documento real, seguirá registrando datos inconsistentes. La implementación debe resolver esto dentro del flujo.
- No encontré la guía local de Next en `node_modules/next/dist/docs/`; si durante la implementación se detectan APIs de navegación o layouts con cambios de Next 16, habrá que verificar la versión instalada antes de tocar patrones sensibles.
